随着英伟达的显卡功能越来越复杂，cuda 也变得越来越难写，大家都要开始学 cutlass 这样的官方工具库了。在 cutlass 的最新版本中，提出了 cute，其在 flash attention 2 中也有比较广泛的应用。不过如 nv 其他的工具库一般，cute 的官方教程写的有些破碎，所以本主要希望能稍稍补全 tutorial 中没讲清的内容，来简单概述一下 cute 中的基本概念。文中描述如果有错漏，请各位大佬指出，感激不尽。

如同官方教程（我选了 tag v3.2.0）所说，下面的这个 gemm 的例子适合作为学习 cute 的开端：

https://github.com/NVIDIA/cutlass/blob/v3.2.0/examples/cute/tutorial/sgemm_nt_1.cu​github.com/NVIDIA/cutlass/blob/v3.2.0/examples/cute/tutorial/sgemm_nt_1.cu

所以我们就照着这个例子来讲吧~

在看代码之前，要简单说几个 cute 里的概念：tensor、layout、shape 和 stride。他们之间有 2 个非常简单的指代关系：

tensor = ptr + layout
layout = shape + stride

就是说，所谓 tensor，就是一个内存指针（ptr，其实是显存，不过为求方便，后面均称内存）和这个指针对应的内存排布的描述信息（layout）；而所谓排布，就是说一个 tensor 的形状（shape），即这个 tensor 是几维的，每一维多大，以及每一维的各个元素之间间隔多大（stride）。啥叫元素间隔多大呢？比如说我们有一个 2 维的 tensor A，形状为 MxN，那么如果他每个元素都是相邻排列的，且为列优先，用 cute 的描述就是形状为 (M, N) ，stride 为 (1, M)，因为列号相同但是行号相邻的 2 个元素之间的距离是 1，行号相同但是列号相邻的 2 个元素之间的距离是 M。

回到样例代码。他在尝试实现一个 CM×N=AM×KBK×NC_{M\times N} = A_{M\times K}B_{K\times N} 的矩阵乘法，其中 M=N=5120，K=4096 都是写死的。算法也非常常规，把 block size 定为 256（16x16），其中的每个 thread 去计算一块 8x8 的 C，也就是一个 block 处理 128x128 的 C。具体操作大致是：

  1. 先在寄存器上申请一块 8x8 的内存作为累加器；
  2. 把一部分 A 和 B 从 global memory 拷贝至 shared memory；
  3. 用 shared memory 中的 A、B 做矩阵乘，把结果存在累加器中；
  4. 循环 2、3 直至这个 128x128 相关的 AB 运算都算完了；
  5. 把累加器加到 C 里对应的 global memory 上。

上述 cute 的这些概念，主要用来帮助我们制定好每个 block，每个 thread 都要访问哪些内存。

根据我们上面提到的关系，假如 A、B、C 都是列优先的，那么他们对应的 cute tensor 就是：

// Represent the full tensors
auto mA = make_tensor(make_gmem_ptr(A), make_shape(M,K), make_stride(Int<1>{}, M));  // (M,K)
auto mB = make_tensor(make_gmem_ptr(B), make_shape(N,K), make_stride(Int<1>{}, N));  // (N,K)
auto mC = make_tensor(make_gmem_ptr(C), make_shape(M,N), make_stride(Int<1>{}, M));  // (M,N)

*其中 Int<1>() 就是表示是个编译期就确定的 1。

因为一个 block 打算每次处理 128x128 的 C，也就是处理 128 个 M 和 128 个 N，那么假设我们在 K 那一维每个 block 每次处理 8 个，我们就有了下面这 3 个常数：

// Define block sizes (static)
auto bM = Int<128>{};
auto bN = Int<128>{};
auto bK = Int<  8>{};

我们也因此可以定下来 kernel 的 dimBlock 和 dimGrid 了：

dim3 dimBlock(size(tC));
dim3 dimGrid(ceil_div(size(M), size(bM)),
             ceil_div(size(N), size(bN)));
gemm_device<<< dimGrid, dimBlock, 0, stream >>>(...)

*注意，这里 dimBlock 是一维的，我们会用 cute 自带的 indexing 来让它发挥 2 维的效果。

上面的这个 bM、bN、bK 相当于每个 block 里每个循环会处理一块 bM x bK 的 A 和一块 bN x bK 的 B。A 和 B 的 shared memory 就应该按这个形状分配，所以它们对应的 cute tensor 为：

// Shared memory buffers
__shared__ TA smemA[bM * bK];
__shared__ TB smemB[bN * bK];
auto sA = make_tensor(make_smem_ptr(smemA), make_layout(make_shape(bM,bK)));  // (BLK_M,BLK_K)
auto sB = make_tensor(make_smem_ptr(smemB), make_layout(make_shape(bN,bK)));  // (BLK_N,BLK_K)

*注意，shared memory 要先手动分配而且分配一维就行，以及 tensor 默认的 stride 是列优先。

下面我们来定一下一个 block 里的 thread 们是怎么干活的，也就是考虑 block 中的 256 个 thread 的 layout。因为 bK=8，所以我们在拷贝的时候这一维最多也就用 8 个 thread，那么 A、B、C 的 thread layout 为：

// Define the thread layouts (static)
auto tA = make_layout(make_shape(Int<32>{}, Int< 8>{}));
auto tB = make_layout(make_shape(Int<32>{}, Int< 8>{}));
auto tC = make_layout(make_shape(Int<16>{}, Int<16>{}));

定义好了这些 tensor 和 layout，我们就可以通过一些函数接口来把每个 thread 对应的 memory 拿出来。先来拿每个 block 对应的 global memory：

// Get the appropriate blocks for this thread block --
// potential for thread block locality
auto blk_shape = make_shape(bM, bN, bK);
auto blk_coord = make_coord(blockIdx.x, blockIdx.y, _);            // (m,n,k)

auto gA = local_tile(mA, blk_shape, blk_coord, Step<_1, _,_1>{});  // (BLK_M,BLK_K,k)
auto gB = local_tile(mB, blk_shape, blk_coord, Step< _,_1,_1>{});  // (BLK_N,BLK_K,k)
auto gC = local_tile(mC, blk_shape, blk_coord, Step<_1,_1, _>{});  // (BLK_M,BLK_N)

这里的操作稍微有点复杂，其实是分了 2 步，第一步是这个 Step 在起作用，他会把 blk_shape 和 blk_coor 中在 Step 中值为 _ 的位置的值都去掉，也就是上面这段等价于是：

auto gA = local_tile(mA, make_shape(bM, bK), make_coord(blockIdx.x, _));  // (BLK_M,BLK_K,k)
auto gB = local_tile(mB, make_shape(bN, bK), make_coord(blockIdx.y, _));  // (BLK_N,BLK_K,k)
auto gC = local_tile(mC, make_shape(bM, bN), make_coord(blockIdx.x, blockIdx.y));  // (BLK_M,BLK_N)

那么剩下这部分是啥意思呢？local_tile 其实是先把 tensor 的形状 reshape 为一个个 tile，然后再取 coord（coordinate，坐标） 位置的 tensor。

具体来说，第一行相当于先把 mA 从 (M, K) reshape 成 (bM, bK, M/bM, K/bK)，然后取 mA[:, :, blockIdx.x, :]（coord 中的 _ 就类似 python 中的 :），得到的 gA 形状为 (bM, bK, K/bK)。类似地，gB 的形状为 (bN, bK, K/bK)，gC 的形状为 (bM, bM)，他们都是当前 block 会访问的 global memory。

*注意，cute 的一个好处是，他会随着这些变形和 indexing 自动调整 stride，不会在过程中做任何内存拷贝，大家可以算一下 gA、gB、gC 的 stride，看看是否和官方 tutorial 里一致。

拿到了每个 block 中要访问的 global memory，那么下一步我们来看每个 thread 要访问的。这里用了一个类似 local_tile 的 api local_partition：

auto tAgA = local_partition(gA, tA, threadIdx.x);  // (THR_M,THR_K,k)
auto tAsA = local_partition(sA, tA, threadIdx.x);  // (THR_M,THR_K)

auto tBgB = local_partition(gB, tB, threadIdx.x);  // (THR_N,THR_K,k)
auto tBsB = local_partition(sB, tB, threadIdx.x);  // (THR_N,THR_K)

和上面的 local_tile 类似，这里也会先做一个变换，会把 threadIdx.x 根据 tA 或 tB 调整成对应的二维 coord，我们不妨称这个转换过的 index 为 (x, y)。然后 local_partition 也是会先按第二个参数 reshape 成一个个 tile，不过这次在取 index 的时候，会在 tile 上取。

具体来说，第一行会把 gA 从 (bM, bK, K/bK) 转成 (tA[0], tA[1], bM/tA[0], bK/tA[1], K/bK) ，然后取 gA[x, y, :, :, :] ，得到 tAgA 的形状为 (bM/tA[0], bK/tA[1], K/bK)，对应当前 thread 要访问的 global memory。类似地，tAsA 的形状为 (bM/tA[0], bK/tA[1])，tBgB 为 (bN/tB[0], bK/tB[1], K/bK)，tBsB 为 (bB/tB[0], bK/tB[1])。其中的 tXgx 为一个线程要访问的 global memory，tXsX 为要访问的 shared memory。

在取每个线程的要访问的 C 时也用了类似的技巧：

// Partition sA (M,K) by the rows of tC
auto tCsA = local_partition(sA, tC, threadIdx.x, Step<_1, X>{});  // (THR_M,BLK_K)
// Partition sB (N,K) by the cols of tC
auto tCsB = local_partition(sB, tC, threadIdx.x, Step< X,_1>{});  // (THR_N,BLK_K)
// Partition gC (M,N) by the tile of tC
auto tCgC = local_partition(gC, tC, threadIdx.x, Step<_1,_1>{});  // (THR_M,THR_N)

大家不妨自己推导一下，看看每个线程需要访问多少 C 呢？尤其是 tCgC 的形状是否是 8x8 呢？

在申请累加器的时候，样例用了一个新的 api：

// Allocate the accumulators -- same size as the projected data
auto tCrC = make_fragment_like(tCgC);  // (THR_M,THR_N)

这里其实就是分配了一个 TC a[8x8] 这样的本地数组，从而能直接使用寄存器。

明确了内存的使用方法，剩下的循环调用 cute 中提供的 gemm 函数就变得非常简单了，就不在这里展开了。

至此，我们通过 gemm 的例子简单过了一下 cute 和内存相关的基本概念和用法，希望对大家学习 cute 有所帮助~
